# Referência da API do Precio

O Precio expõe um único endpoint no backend, `api.php`, responsável pela autenticação e pela busca de produtos. É uma API baseada em ações (estilo RPC), não uma API REST de recursos — toda requisição é um `POST` para a mesma URL, com um campo `action` indicando ao servidor o que fazer.

Esta referência foi gerada a partir da implementação atual (`api.php`, `includes/auth.php`, `includes/db.php`).

---

## URL Base

```
POST /api.php
```

Não há versionamento nem suposição de um hostname separado — o frontend (`index.php`, `home.php`) chama esse endpoint com um caminho relativo, ou seja, ele foi projetado para uso na mesma origem (same-origin).

## Formato da Requisição

- **Método:** apenas `POST`. Qualquer outro método ou a ausência de `action` cai no fluxo padrão de "ação inválida".
- **Corpo:** `multipart/form-data` (o frontend monta as requisições com `FormData` + `fetch`), mas como o PHP lê a partir de `$_POST`, corpos padrão `application/x-www-form-urlencoded` também funcionam.
- **Campo de despacho:** toda requisição deve incluir `action` com um dos valores: `login`, `register`, `logout`, `search`.

```js
// Padrão usado em todo o frontend
const fd = new FormData();
fd.append('action', 'search');
fd.append('busca', 'rtx 4070');
const res = await fetch('api.php', { method: 'POST', body: fd });
const json = await res.json();
```

## Formato da Resposta

Toda resposta tem `Content-Type: application/json` dentro de um envelope no formato:

```json
{ "success": true, "...": "..." }
```

ou, em caso de falha:

```json
{ "success": false, "message": "Mensagem de erro em português." }
```

**Importante:** a API sempre retorna HTTP `200 OK`, inclusive em erros de validação, falhas de autenticação e ações inválidas. Não há uso significativo de códigos de status HTTP — quem consome a API deve verificar o booleano `success`, não o status HTTP. As mensagens de erro/validação já vêm em português (pt-BR), pois são exibidas diretamente na interface.

## Autenticação

O Precio usa sessões nativas do PHP (baseadas em cookie), não tokens nem chaves de API.

- `login` e `register` chamam `loginUser()`, que regenera o ID da sessão e armazena `user_id`, `user_name`, `user_email` em `$_SESSION`.
- `logout` chama `logoutUser()`, que limpa e destrói a sessão.
- `search` chama `isLoggedIn()` e exige uma sessão ativa; não há como autenticar uma requisição de `search` sem antes fazer login por essa mesma API (não existem tokens de portador nem chaves de API).

Como a autenticação é baseada em sessão/cookie, essa API é voltada para o próprio frontend da aplicação, não para integrações servidor-a-servidor de terceiros.

## Endpoints

### `action=login`

Autentica um usuário e inicia uma sessão.

| Campo | Tipo | Obrigatório | Observações |
|---|---|---|---|
| `email` | string | sim | Recebe `trim()` no servidor |
| `password` | string | sim | **Não** recebe `trim()` — espaços são tratados como significativos |

**Sucesso (200):**
```json
{ "success": true, "redirect": "home.php" }
```

**Falha (200):**
```json
{ "success": false, "message": "Preencha todos os campos." }
```
```json
{ "success": false, "message": "E-mail ou senha incorretos." }
```

Observação: e-mail inválido e senha incorreta retornam a *mesma* mensagem genérica ("E-mail ou senha incorretos.") — a API deliberadamente não revela se o e-mail existe ou não.

---

### `action=register`

Cria um novo usuário e já efetua o login automaticamente.

| Campo | Tipo | Obrigatório | Observações |
|---|---|---|---|
| `name` | string | sim | Recebe `trim()` |
| `email` | string | sim | Recebe `trim()`; precisa passar em `FILTER_VALIDATE_EMAIL`; precisa ser único |
| `password` | string | sim | Não recebe `trim()`; mínimo de 6 caracteres |
| `confirm` | string | sim | Precisa ser idêntico a `password` |

**Sucesso (200):**
```json
{ "success": true, "redirect": "home.php" }
```

**Falha (200)** — uma das opções abaixo:
```json
{ "success": false, "message": "Preencha todos os campos." }
```
```json
{ "success": false, "message": "E-mail inválido." }
```
```json
{ "success": false, "message": "A senha deve ter ao menos 6 caracteres." }
```
```json
{ "success": false, "message": "As senhas não coincidem." }
```
```json
{ "success": false, "message": "Este e-mail já está cadastrado." }
```

A ordem de validação importa: presença dos campos → formato do e-mail → tamanho da senha → confirmação da senha → unicidade do e-mail. A primeira verificação que falhar interrompe a requisição. As senhas são armazenadas com `password_hash()` (bcrypt).

---

### `action=logout`

Destrói a sessão atual. Sem parâmetros.

**Resposta (200):**
```json
{ "success": true, "redirect": "index.php" }
```

---

### `action=search` 🔒 requer login

Busca um produto em seis varejistas brasileiros em paralelo e retorna uma lista mesclada, ordenada por preço.

| Campo | Tipo | Obrigatório | Observações |
|---|---|---|---|
| `busca` | string | sim | Recebe `trim()`. Precisa ter entre 2 e 120 caracteres (medido com `mb_strlen`) |

**Não autenticado:**
```json
{ "success": false, "message": "Sessão expirada. Faça login novamente." }
```

**Falhas de validação:**
```json
{ "success": false, "message": "Digite pelo menos 2 caracteres para pesquisar." }
```
```json
{ "success": false, "message": "Termo de pesquisa muito longo." }
```

**Sucesso (200):**
```json
{
  "success": true,
  "tempo": 1.84,
  "do_cache": false,
  "data": {
    "total": 24,
    "produtos": [
      {
        "nome": "Placa de Vídeo RTX 4070 Super 12GB",
        "sku": "GPU-4070S-12",
        "preco": 4299.99,
        "preco_orig": 4799.90,
        "desconto": 10.4,
        "url": "https://www.pichau.com.br/placa-de-video-rtx-4070-super-12gb",
        "imagem": "https://media.pichau.com.br/.../rtx4070super.jpg",
        "em_estoque": true,
        "loja": "Pichau"
      }
    ],
    "fonte": "Pichau, Mercado Livre, KaBuM!, Amazon, Magazine Luiza e Terabyte Shop",
    "lojas": { "Pichau": 6, "KaBuM!": 9, "Amazon": 4 },
    "aviso": ""
  }
}
```

#### Objeto `data`

| Campo | Tipo | Descrição |
|---|---|---|
| `total` | int | Número de produtos em `produtos` (após filtragem, com teto de 36) |
| `produtos` | array | Lista de produtos, ordenada por `preco` crescente (itens sem preço ficam por último) |
| `fonte` | string | Rótulo estático nomeando os cinco varejistas consultados |
| `lojas` | object | Mapa de nome da loja → quantidade de produtos nesse conjunto de resultados |
| `aviso` | string | Mensagem de aviso, por exemplo quando os dados de uma loja vieram de um scraper alternativo (fallback) em vez da API principal. String vazia quando não há nada a reportar |

#### Objeto de produto (`produtos[i]`)

| Campo | Tipo | Descrição |
|---|---|---|
| `nome` | string | Nome do produto |
| `sku` | string | SKU quando disponível (só é preenchido no caminho via GraphQL da Pichau); string vazia nos demais casos |
| `preco` | float | Preço atual/final em reais (BRL), sem símbolo de moeda |
| `preco_orig` | float\|null | Preço original (antes do desconto), preenchido apenas quando diferente de `preco` |
| `desconto` | float\|null | Percentual de desconto, quando conhecido |
| `url` | string | URL direta do produto no site do varejista |
| `imagem` | string | URL da imagem do produto |
| `em_estoque` | bool | Se o item está em estoque (assume `true` por padrão para fontes obtidas via scraping que não expõem status de estoque) |
| `loja` | string | Um dos valores: `"Pichau"`, `"Mercado Livre"`, `"KaBuM!"`, `"Amazon"`, `"Magazine Luiza"`, `"Terabyte Shop"` |

#### Sobre a origem dos dados do Mercado Livre

Diferente das outras lojas, o Mercado Livre não é obtido via proxy/parsing de HTML — é consultado diretamente pela **API pública oficial** (`GET https://api.mercadolibre.com/sites/MLB/search`), que devolve JSON estruturado. Isso torna essa fonte a mais estável e menos sujeita a quebrar com mudanças de layout.

Um `client_id`/`client_secret` (gerados em [developers.mercadolivre.com.br](https://developers.mercadolivre.com.br)) podem ser configurados no painel admin (Configurações). Com eles, o back-end obtém um token de aplicação via OAuth `client_credentials` (cacheado por ~6h) e o envia como `Authorization: Bearer`, elevando o limite de requisições. Sem credenciais, a busca funciona do mesmo jeito usando o endpoint público sem autenticação, só que com um limite de requisições por IP mais baixo.

**Fallback contra rate limit:** quando a API oficial não devolve nenhum produto (rate limit HTTP 429, bloqueio temporário, etc.), o sistema cai automaticamente para um scraping via proxy Jina Reader da página pública de busca (`lista.mercadolivre.com.br`) — a mesma estratégia já usada para Pichau/KaBuM!/Amazon quando a fonte principal falha. Isso significa que mesmo sem `client_id`/`client_secret`, o Mercado Livre tende a continuar aparecendo nos resultados na maioria das buscas, só que com uma origem diferente por trás dos panos.

#### Observações sobre o comportamento da busca

- **Cache:** os resultados ficam em cache em arquivo por 10 minutos, indexados pelo termo de busca em minúsculas. Um acerto de cache (cache hit) retorna instantaneamente com `do_cache: true` e pula toda a etapa de scraping externo. Apenas buscas bem-sucedidas com pelo menos um produto são armazenadas em cache — resultados vazios nunca são cacheados, então uma busca sem retorno tenta de novo do zero na próxima vez.
- **Origem dos dados:** a Pichau é consultada primeiro via sua API GraphQL; se isso falhar ou for bloqueado (desafio do Cloudflare, resposta de erro), o sistema recorre a um scraping via proxy do Jina Reader. O Mercado Livre e as lojas VTEX cadastradas (ver abaixo) são consultados diretamente via suas APIs públicas (JSON, sem proxy). KaBuM!, Amazon, Magazine Luiza e Terabyte Shop são sempre obtidos via proxy do Jina Reader e interpretados a partir do Markdown retornado.

**Formato real confirmado (calibrado em 05/07/2026 com uma resposta real da Terabyte Shop):** o Jina Reader embrulha imagem + nome + avaliação + selos + preço num único link markdown — `[![Image N](url_imagem) Nome ★★★★★4.8(42) 🔥 em promoção Marca De: R$ ORIGINAL R$ ATUAL 12x R$ parcela](url_produto)` — não em dois links separados como uma versão anterior deste parser assumia. O parser genérico usado por Magazine Luiza, Terabyte Shop e pelo fallback do Mercado Livre foi corrigido e testado contra essa estrutura real (preço com/sem desconto, produtos esgotados com preço único, nomes sem avaliação, entidades HTML como `&quot;`).

**Limitação conhecida — Magazine Luiza:** o WAF do Magalu costuma bloquear o IP do proxy Jina Reader com HTTP 403 ("Por favor, tente novamente em 1 minuto"), impedindo a coleta independentemente do parser estar correto. Isso não é um bug do parser — é um bloqueio anti-bot no lado do Magalu contra o próprio proxy. O sistema já trata esse caso sem quebrar (retorna 0 produtos dessa loja naquela busca).

#### Lojas VTEX (opcional, configurável pelo painel admin)

Redes que rodam na plataforma VTEX (ex.: Centauro, Fast Shop, Lojas Colombo) expõem um endpoint público de catálogo, sem necessidade de chave de API:

```
GET https://{dominio-da-loja}/api/catalog_system/pub/products/search?ft={termo}
```

Essas lojas são cadastradas dinamicamente pelo painel admin (Configurações → Lojas VTEX, informando nome + domínio) e entram automaticamente na mesma leva de requisições paralelas do `search`. O campo `fonte` da resposta é montado dinamicamente e passa a incluir o nome de cada loja VTEX ativa. Não há lista fixa de lojas VTEX no código — cabe ao administrador testar o domínio (`.../api/catalog_system/pub/products/search?ft=teste` deve responder com uma lista JSON) antes de cadastrar.
- **Filtragem por relevância:** os resultados são filtrados em relação à busca usando correspondência de tokens e regras específicas por categoria (ex.: buscas por placa de vídeo excluem PCs prontos, monitores e teclados, a menos que a própria busca mencione esses termos).
- **Teto de resultados:** até 36 produtos são retornados por busca, ordenados por preço crescente.
- **Tempo:** `tempo` é a duração da busca no servidor, em segundos (float, arredondado a 2 casas decimais). Em um acerto de cache, esse valor reflete a duração da busca *original*, não o tempo (quase nulo) da consulta ao cache.

## Resumo do Tratamento de Erros

| Cenário | Status HTTP | `success` | Observações |
|---|---|---|---|
| `action` ausente/inválida | 200 | `false` | `message: "Ação inválida."` |
| Campos obrigatórios ausentes | 200 | `false` | Mensagem em pt-BR específica do campo |
| Não autenticado (search) | 200 | `false` | `message: "Sessão expirada. Faça login novamente."` |
| Falha de conexão com o banco | 200 | n/a | Retorna `{"error": "Erro de conexão com o banco de dados."}` — **não** segue o envelope padrão; ocorre em `getDB()`, antes do fluxo normal de resposta |

Como os códigos de status não são usados para sinalizar o resultado, o código cliente deve sempre interpretar o corpo JSON e decidir com base em `success` (ou na presença de `error`, para o caso de falha de banco de dados acima).

## Limites de Requisição (Rate Limits)

Não há limitação explícita de taxa (rate limiting) nem throttling nessa API. O cache em arquivo de 10 minutos no `search` é o principal mecanismo que evita que buscas idênticas repetidas disparem novos scrapings contra os sites dos varejistas.

## Paginação

Nenhum dos endpoints faz paginação. O `search` retorna o conjunto completo de resultados (com teto de 36 itens) em uma única resposta.

## Referência Rápida

| Ação | Requer autenticação | Método | Parâmetros principais |
|---|---|---|---|
| `login` | não | POST | `email`, `password` |
| `register` | não | POST | `name`, `email`, `password`, `confirm` |
| `logout` | não | POST | — |
| `search` | sim (sessão) | POST | `busca` |

## Exemplo: cURL

```bash
# Login (mantém os cookies para a chamada de search, que exige sessão)
curl -c cookies.txt -X POST https://seu-host/api.php \
  -F "action=login" -F "email=usuario@exemplo.com" -F "password=senha123"

# Busca usando o cookie de sessão
curl -b cookies.txt -X POST https://seu-host/api.php \
  -F "action=search" -F "busca=rtx 4070"
```
